# 首页动态化项目规划

> 依据：项目说明书 §3.1、§4.1、§5.2、§6.2（8）；权限说明 §4.3  
> 目标：官网 onMounted 请求最新配置，实现由后台修改的图文在官网实时展示

---

## 1. 现状与目标

### 1.1 现状

| 模块 | 官网 (`web/website`) | 管理后台 (`web/admin`) | 后端 API |
|------|----------------|------------------|----------|
| **首屏 Hero** | 硬编码标题、背景图、登录链接 | HeroConfig 可编辑 title/subtitle/background_image/cta | ✅ GET/PUT /homepage/config |
| **核心理念** | 硬编码视频、文案 | PhilosophyConfig 可编辑 title、items[] | ✅ |
| **设施展示** | 硬编码视频、文案 | FacilitiesConfig 可编辑 title、items[] | ✅ |
| **为什么选择我们** | 硬编码 4 面板 | WhyChooseUsConfig 可编辑 title、items[] | ✅ |
| **公告/资讯** | 静态 newsData | 无配置入口 | 可接 articles 接口 |
| **家长感受** | 硬编码视频、引用 | 无配置入口 | 可扩展 config |
| **社交矩阵** | 硬编码 4 卡片 | 无配置入口 | 可扩展 config |
| **联系方式** | 硬编码地址、电话 | ContactConfig 可编辑 | ✅ |

### 1.2 目标

- 官网各区块在 `onMounted` 时请求 `GET /api/homepage/config`，用返回数据渲染
- 配置为空或接口失败时，使用现有硬编码作为兜底，保证页面可访问
- 管理后台已支持 5 个区块配置，官网需与之一一对应

---

## 2. 配置键与数据结构约定

与 admin 端保持一致，官网读取的 `config_key` 与数据结构如下：

| config_key | 用途 | 数据结构（config_value 为 JSON 字符串） |
|------------|------|----------------------------------------|
| `hero` | 首屏 Banner | `{ title, subtitle, background_image, cta_text, cta_link }` |
| `philosophy` | 核心理念 | `{ title, items: [{ title, description, icon }] }` |
| `facilities` | 设施展示 | `{ title, items: [{ name, description, image }] }` |
| `why_choose_us` | 为什么选择我们 | `{ title, items: [{ title, description, icon }] }` |
| `contact` | 联系方式 | `{ address, phone, email, wechat, working_hours, latitude, longitude }` |

**说明**：`GET /api/homepage/config` 返回 `{ success, data: { [config_key]: config_value } }`。`config_value` 为字符串时需 `JSON.parse` 解析为对象。

## 3. 实施阶段与任务拆解

### 阶段一：基础设施（约 0.5 天）

| 序号 | 任务 | 说明 | 产出 |
|------|------|------|------|
| 1.1 | 官网 API 模块 | 在 `web/website/src/api/` 下新增 `homepage.ts`，封装 `getHomepageConfig()`，请求 `GET /api/homepage/config` | `web/website/src/api/homepage.ts` |
| 1.2 | 配置类型定义 | 定义 `HeroConfig`、`PhilosophyConfig` 等 TypeScript 类型，与 admin 端结构一致 | `web/website/src/types/homepage.ts` |
| 1.3 | 全局配置 Store | 使用 Pinia 或 provide/inject 存储首页配置，供各组件消费；支持 loading、error 状态 | `web/website/src/stores/homepage.ts` 或 `web/website/src/composables/useHomepageConfig.ts` |
| 1.4 | Vite 代理配置 | 开发环境下将 `/api` 代理到后端（若尚未配置） | `web/website/vite.config.ts` |

### 阶段二：区块动态化（约 1.5 天）

按优先级依次改造各区块，每个区块需：**拉取配置 → 有则用配置渲染，无则用默认值**。

| 序号 | 区块 | 配置键 | 改造要点 | 默认兜底 |
|------|------|--------|----------|----------|
| 2.1 | **HeroSection** | hero | 主标题、副标题、背景图、CTA 按钮文字与链接 | 当前硬编码文案与图 |
| 2.2 | **ContactConfig → SiteFooter** | contact | 地址、电话、邮箱、微信、工作时间 | 当前硬编码 |
| 2.3 | **EthosSection** | philosophy | 区块标题、左侧视频（可扩展为 config 中的 video_url）、右侧文案 items | 当前硬编码 |
| 2.4 | **FacilitySection** | facilities | 区块标题、文案、视频/图片（可扩展） | 当前硬编码 |
| 2.5 | **WhyChooseUsSection** | why_choose_us | 区块标题、4 面板改为 `v-for` 渲染 items，支持动态数量 | 当前 4 面板硬编码 |

### 阶段三：扩展与优化（约 0.5 天）

| 序号 | 任务 | 说明 | 优先级 |
|------|------|------|--------|
| 3.1 | 公告/资讯动态化 | 从 `GET /api/articles?status=published&limit=6` 拉取，替代静态 `newsData`；需确认 articles 接口与权限 | 中 |
| 3.2 | 家长感受、社交矩阵 | 若需后台可配置，扩展 `homepage_config` 新增 key（如 `testimonial`、`social_matrix`），并在 admin 增加对应配置页 | 低 |
| 3.3 | 配置缓存 | 官网为公开页，可对 `GET /api/homepage/config` 做短期缓存（如 5 分钟），减少 DB 压力 | 低 |
| 3.4 | 错误与空状态 | 接口失败时静默降级为默认内容，不阻塞首屏；可选在控制台输出 warning | 中 |

## 4. 技术方案要点

### 4.1 数据流

```
App.vue onMounted
    → fetchHomepageConfig()  // 调用 GET /api/homepage/config
    → 解析 config_value（JSON.parse）
    → 写入 store / provide
    → 各区块组件从 store 读取对应 key，有则渲染，无则用默认
```

### 4.2 兜底策略

- 每个区块保留一份 `defaultConfig`，与当前硬编码一致
- 渲染时：`const config = store.hero ?? defaultHero`
- 接口失败时：store 为空，全部使用 default

### 4.3 与 admin 的 config_key 对齐

| Admin 组件 | 请求路径 | config_key | 官网对应组件 |
|------------|----------|------------|--------------|
| HeroConfig | /homepage/config/hero | hero | HeroSection |
| PhilosophyConfig | /homepage/config/philosophy | philosophy | EthosSection |
| FacilitiesConfig | /homepage/config/facilities | facilities | FacilitySection |
| WhyChooseUsConfig | /homepage/config/why_choose_us | why_choose_us | WhyChooseUsSection |
| ContactConfig | /homepage/config/contact | contact | SiteFooter |

## 5. 后端需确认事项

| 项目 | 说明 |
|------|------|
| **GET /api/homepage/config 返回格式** | 当前返回 `{ config_key: config_value }`，config_value 为 DB 原始值（JSON 字符串）。官网需统一 `JSON.parse`，或后端在 getAllConfig 中解析后返回对象。 |
| **跨域** | 官网与 API 若不同域，需后端 CORS 允许官网域名。 |
| **Base URL** | 官网需配置 `VITE_API_BASE_URL` 指向后端（如 `https://lexingronghe.com/api`），开发时可用 proxy。 |

---

## 6. 验收标准

1. 管理员在后台修改「首屏 Banner」「联系方式」等配置并保存后，刷新官网，对应区块展示与配置一致。
2. 未配置或配置为空时，官网展示默认内容，无报错、无空白。
3. 接口超时或 5xx 时，官网降级为默认内容，不白屏。
4. 与项目说明书 §3.1 验收标准一致：*管理员可登录后台完成活动页编辑与首页素材替换，前端展示与配置一致。*

---

## 7. 实施顺序建议

```
1. 阶段一（基础设施）→ 2. 阶段二 2.1 Hero + 2.2 Contact（先打通最小闭环）
   → 3. 联调验证（后台改配置 → 官网刷新可见）
   → 4. 阶段二剩余区块（Ethos、Facility、WhyChooseUs）
   → 5. 阶段三（按需）
```

---

## 8. 风险与依赖

| 风险 | 缓解 |
|------|------|
| 官网与 admin 的 config 结构不一致 | 严格按 admin 现有字段定义类型，并在文档中固化 |
| 后端 config_value 存的是字符串，前端解析失败 | 加 try/catch，解析失败时用默认值 |
| 官网未配置 API 代理导致跨域 | 在 vite.config 中配置 proxy，生产通过 Nginx 或同域部署 |
